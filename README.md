# ♛ Beggars to Crowns — AI RPG

Written with the help of Claude Sonnet 4.6.

A terminal-based multi-player RPG powered by the HuggingFace Inference API.
Each player has their own isolated story context.

---

## Requirements

- [Rust](https://rustup.rs/) (stable, 1.70+)
- A free HuggingFace account + API token

---

## Setup

### 1. Get a HuggingFace API token (free)

1. Sign up at https://huggingface.co
2. Go to **Settings → Access Tokens**
3. Create a token with **"Read"** permission (free tier)
4. The free tier gives generous rate limits on Mistral-7B

### 2. Build the game

```bash
cd beggars_to_crowns
cargo build --release
```

### 3. Run

```bash
cargo run --release
```

---

## Gameplay

- At launch, enter the number of players and each player's name.
- Each player gets their own opening scene and isolated AI memory.
- Players take turns. On your turn, just type your action or dialogue.

### Commands (any turn)
| Command | Effect |
|---------|--------|
| `stats` | Show prompt count + character count per player |
| `switch <name>` | Jump to a specific player's turn |
| `quit` / `exit` | End the game and show final stats |

---

## Customizing the Story Prompt

Open `src/main.rs` and edit the `STORY_PROMPT` constant at the top:

```rust
const STORY_PROMPT: &str = "\
The 'Beggars to Crowns' Challenge\n\
...your custom settings here...\
";
```

Then rebuild: `cargo build --release`

## Model

Uses **`meta-llama/Llama-3.1-8B-Instruct`** — one of the best open models for
creative storytelling with a generous free HuggingFace Inference tier.

To swap models, change `HF_MODEL` in `main.rs`. Any HF model supporting the
`/v1/chat/completions` endpoint works.

## FAQ

1. Does this code make AI instances for each character?
  Yes, effectively. The code does not spawn separate background processes, but it isolates the memory for each player, creating separate, parallel universes.

  Here is how it works:

  In setup_players, it creates a HashMap<String, PlayerSession>.
  Each PlayerSession contains its own history: Vec<Message>.
  When a player takes a turn in game_loop, the code fetches only that specific player's history (sessions[&name].history.clone()) and sends it to the HuggingFace API.
  This means Player 1 and Player 2 are playing the same scenario, but their choices do not affect each other. Player A's AI has no idea what Player B is doing.

2. What does the AI see initially, and what does it not see?
  What the AI DOES see initially: Before the player types their first command, the AI receives exactly two messages (found in the PlayerSession::new and opening_scene functions):

  A system message: This is the massive set of instructions generated by build_system_prompt(story, &common_rules). It contains the scenario's background, setting details, character inventory, active rules, and win conditions.
  The first user message (hardcoded):
  "Begin the game. Deliver the opening scene exactly as written in your instructions. Describe the environment vividly with full sensory detail and present the first situation and choices. Address the player as 'you'."

  What the AI does NOT see: Because of how the code parses data, the AI is completely blind to several things:

  The Player's Name: The name variable is used to label the turns in the local terminal UI and stats menu, but it is never actually injected into the Message history. The AI only knows the player as "you" or by whatever name the scenario defaults to (like "CleanSweep").
  Other Players: As mentioned above, it cannot see the prompts or responses from other players' sessions.
  System Commands: If a player types quit, stats, or switch <name>, the code intercepts these commands in the game_loop and executes them locally. These words are never sent to the AI.
  Empty inputs: Hitting "Enter" without typing anything skips the turn locally and is not sent to the AI.
  The UI Menus: The AI does not see the help menus, the scenario selection screen, or the "Beggars to Crowns" ASCII header.
